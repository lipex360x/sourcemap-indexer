## What

Order pending files by their dependency graph before enriching, so that when a dependent file is processed, its direct imports are already enriched in SQLite with `purpose` filled. This makes `--with-context` useful on the very first `enrich` run against a fresh index — no second pass required.

## Why

- Single-pass enrichment is the expected behaviour of `--with-context` — the two-pass workaround is a gambiarra.
- Topological ordering is the standard approach in build tools (cargo, bazel, make) and is a natural fit: leaves enriched first, dependents after.
- Result blocks in SQLite are committed synchronously after each `client.enrich()` call, so by the time the dependent reaches the LLM, the import's `purpose` is already queryable.

## Acceptance criteria

- [ ] `_topologically_ordered(items, extractors, root) -> list[Item]` helper in `application/enrich.py` — Kahn's algorithm, stable within tiers
- [ ] Cycles detected and degraded gracefully: items in a cycle are processed without context (current behaviour) — no crash, no infinite loop
- [ ] Items whose language has no extractor (TS/JS today) land in tier 0 with arbitrary order between them
- [ ] Items with `SyntaxError` in content have `[]` deps → treated as leaves (tier 0)
- [ ] `--file`, `--layer`, `--language` filters still work; topo-sort ordering applies only within the filtered set
- [ ] Unit tests: leaf → dependent ordering, cycle degradation, missing extractor falls to tier 0, single-item input is no-op
- [ ] Integration test: fresh base + `run_enrich(with_context=True)` on two Python files (one imports the other) produces non-empty context block in the dependent's LLM prompt
- [ ] README updated — note that `--with-context` now works in a single pass
